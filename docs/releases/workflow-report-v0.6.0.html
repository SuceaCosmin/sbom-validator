  2. Three parallel TDD tracks delivered cleanly with zero implementation rework.
    The planner's parallel track structure (3.D1/3.E1/3.F1 written in parallel, then 3.D2/3.E2/3.F2 implemented in parallel) worked as designed. All 107 new tests passed on first implementation attempt for each track. This is the largest feature implemented in a single cycle (new module, 3 new test files, 6 fixture files, 1 schema file) with zero green-to-red regression in existing tests.
  3. G4 correctly elevated an ADR contract violation to Architect level.
    The reviewer recognized that M-01 (envelope schema approach) was not a simple code quality issue but an architectural contract violation requiring formal ADR amendment. Rather than treating it as a developer fix, the reviewer correctly classified it as MAJOR requiring Architect dispatch. This preserved the integrity of the ADR as the authoritative design record. The outcome — ADR-010 Amendment 1 — ensures the deviation is formally documented and D2 deferral is traceable to an Architect decision.
  4. CycloneDX DeprecationWarning debt eliminated as a bonus fix.
    The 20 DeprecationWarnings from missing auxiliary CycloneDX schemas had been accumulating since v0.3.0. The CI Ops agent identified and resolved the root cause during G6 stabilization in this cycle. The fix — bundling spdx.schema.json and jsf-0.82.schema.json — is low-risk and delivers permanent improvement: 0 warnings in the test suite going forward, strengthening the "no network calls at runtime" invariant. This was a pre-existing issue resolved opportunistically rather than a new regression.
  5. S-M-02 (registry guard) caught a genuine runtime-network-call risk at G5.
    The Draft202012Validator constructed without registry=Registry() would have attempted external HTTP fetches on any validation failure that triggered the else: $ref AnyClass branch in the SPDX 3.x schema. This would have silently violated the architectural invariant that no network calls occur at runtime, and would have produced non-deterministic behavior in air-gapped environments. G5 caught this before release — an example of the security gate providing value beyond supply-chain concerns.

5. Top Workflow Gaps

  1. Developer deviated from ADR-010 without prior Architect consultation (root cause of G4 M-01).
    Root cause: The developer encountered a structural incompatibility in the bundled SPDX 3.0.1 schema (the shacl2code-generated else: $ref AnyClass branch rejects minimal @context-only root documents, which are valid SPDX 3.x JSON-LD). Instead of raising this to the Architect to amend ADR-010 before implementing the workaround, the developer implemented an envelope-only validation approach and committed it. ADR-010 was not updated to reflect this deviation. The G4 reviewer correctly found this as M-01.
    Impact: Medium. Required a retroactive Architect dispatch (Amendment 1 to ADR-010), a developer fix dispatch (removing dead constant M-02, which resulted from the same deviation), and a formal deferral (D2) with risk documentation. No functional defect shipped, but the ADR was temporarily stale with respect to the implementation. The rework cost — review + Architect amendment + fix commit — was avoidable if the deviation had been surfaced pre-implementation.
    Fix: Add to developer.md: "If implementing a module governed by an ADR and the implementation requires deviating from any decision in that ADR, stop, raise the deviation to the Architect, and obtain an ADR amendment before committing." Consider adding an explicit checklist item to the developer's pre-commit gate: "Does this implementation match all decisions in the governing ADR(s)?"
  2. Missing auxiliary schemas in CycloneDX bundle undetected for 3 releases (v0.3.0–v0.5.0).
    Root cause: When CycloneDX multi-version support was added in v0.3.0, the primary CycloneDX JSON schemas were bundled but the auxiliary schemas they reference via $ref were not. The jsonschema library fell back to remote reference resolution (emitting DeprecationWarnings) rather than raising a hard error. Because the fallback succeeded (network was available in dev environments), the missing schemas were not caught by CI and the warnings were apparently not treated as blocking. This pattern persisted across v0.3.0, v0.4.0, and v0.5.0.
    Impact: Low (no functional defect in networked environments). High in air-gapped environments where the remote fetch would fail, causing CycloneDX validation to fail on any schema with a $ref to a missing auxiliary. The "no network calls at runtime" architectural invariant was being silently violated for every CycloneDX validation.
    Fix: Add a schema self-test step to CI: after loading all bundled schemas, assert that all $ref URIs are resolvable within the local bundle. This would have caught the missing auxiliary schemas in v0.3.0 before they became a multi-release debt.
  3. 2-session delivery creates coordination overhead and session-boundary risk.
    Root cause: Session 1 hit the token limit during G3 (after test writing but before all implementations were complete). This is inherent to the feature size — a new module with 3 parallel test tracks, 6 fixture files, and 2 ADR documents is a large context load. The boundary occurred at a well-defined point (all tests written, no implementations started), which minimized reconstruction cost. However, the session boundary between G3 (test writing) and G3 (implementations) required re-reading TASKS tracker state and re-establishing context for each implementation agent.
    Impact: Low-Medium. No tasks were duplicated or skipped. The TASKS tracker accurately reflected state at session end. Session 2 resumed correctly. However, there is a structural risk: if the TASKS tracker had not been kept current at session 1 end, Session 2 context reconstruction would have been unreliable.
    Fix: For features where G3 is expected to span multiple sessions (3+ parallel TDD tracks + implementations = high context load), the planner should explicitly schedule a "session checkpoint" task: write a TASKS tracker update with current gate status and all completed/pending subagents before the context limit is approached. This makes the session boundary an intentional process step rather than an emergency stop.

Overall Trend Verdict

STABLE WITH ACTIVE REWORK PLATEAU

The sbom-validator workflow continues to demonstrate strong structural health: zero CI lint/format failures for five consecutive releases, 100% gate pass rate sustained, and consistently high coverage (95–96%). The structural improvements from v0.2.2 (pre-commit hooks) and v0.5.0 (documentation completeness at G7) are holding.

However, the rework ratio — measured by G4 fix dispatches required per release — has not declined. v0.5.0: 2 fix dispatches (functional bug + documentation). v0.6.0: 2 fix dispatches (ADR amendment + code fix). The failure modes differ each release (multi-parser consistency, then documentation gaps, now ADR communication), which means single-issue fixes are not converging to zero rework. Each cycle introduces a new upstream gap type.

The primary lever for the next improvement phase is closing the developer-to-ADR communication loop: requiring developers to seek Architect approval for any ADR deviation before committing, rather than discovering the deviation at G4. If this change is applied to developer.md, the M-01 pattern should not recur. Combined with the schema self-test CI gate (Recommendation 2), the structural integrity of the "no network calls" invariant will be enforced mechanically rather than relying on G5 to catch it.

The active deferral D2 (envelope-only schema validation) is the most significant technical debt introduced in this release. It is formally documented with risk acceptance, but it should be scheduled for resolution in v0.7.0 to prevent it from becoming a multi-release debt like the CycloneDX auxiliary schema gap that persisted from v0.3.0 to v0.6.0.
