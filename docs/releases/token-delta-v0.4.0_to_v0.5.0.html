New Friction Points Identified in v0.5.0

  • Verification-only releases amplify review gate costs: When no new feature branch is created and all work is already merged to develop, the G4 reviewer must examine a cumulative diff (14 commits) rather than a focused feature branch. This inflated the reviewer context window by ~5–6x vs v0.4.0, making G4 the dominant gate instead of G3. Future verification-only releases should scope the reviewer's input to the diff since the last release tag, not the full session history.
  • Cross-parser consistency gap (F-01 root cause): DEPENDENCY_OF was added to the JSON/YAML SPDX parsers but not to spdx_tv_parser.py. The original feature implementation had no checklist item requiring "apply fix to all SPDX serialization variants." This is a structural pattern risk — any parser-level fix can have this omission.
  • Documentation written across multiple files without a consolidation pass: F-02 (CHANGELOG), F-03 (requirements.md), F-04 (user-guide.md) were all documentation consistency gaps. All three could have been caught in a single proactive documentation-writer pass before merging to develop. Instead they were caught reactively in G4, each requiring a separate fix dispatch with a full context reload.
  • G8 checklist scope expansion without cost calibration: The release manager's checklist now includes JSON output key stability verification (new in v0.5.0 due to the category and tool_version fields). This expanded the G8 dispatch cost from ~5,000–8,000 (v0.4.0) to 42,270. The checklist expansion is appropriate, but the cost increase was not anticipated in the v0.4.0 forecast.
  • Behavioral change releases (FR-07 removal) require higher G5 security review investment: The v0.4.0 G5 forecast assumed future security reviews would remain at ~10,000–15,000 tokens. A release that removes a validation check requires substantive scope analysis, not just an input-validation audit. Security review cost scaling should factor in whether the release includes any behavioral change to the validation pipeline.

Net Efficiency Verdict

FLAT (structurally expected)

Raw token spend for v0.5.0 (297,770 measured) is approximately 29% above the v0.4.0 midpoint estimate (~230,000). However, this increase is structurally expected given the release pattern: v0.5.0 is a verification-only release (no new feature branches), which eliminates the large G3 TDD Build cost but shifts the token burden entirely to the G4 reviewer, G5 security reviewer, and G8 release manager. These agents each require large context windows proportional to the accumulated diff on develop since the last release tag.

The rework ratio increase (5–8% vs 4–6%) reflects one genuine defect (F-01: DEPENDENCY_OF) that was missed during original feature development. This is the primary actionable finding from this release: cross-parser consistency checking must be a mandatory gate item for any parser-level fix.

Confidence in v0.5.0 data is higher than v0.4.0 — six of seven agent values are measured rather than estimated. This is a structural improvement that benefits forecast accuracy for future releases.
