Key Observations — v0.4.0 Specifics

  • Single session, fully orchestrated (169 turns, 52 minutes): v0.4.0 completed entirely within one session with no boundary reload cost. This is the most cache-efficient pattern possible: context stays warm throughout, resulting in 98.1% cache efficiency.
  • Write-heavy release: Write (21 calls, 45.4% of output) dominated over Edit (32 calls, 17.6%), which is characteristic of new-parser releases where substantial new file content must be created rather than modifying existing files.
  • Edit call count exceeded Write call count (32 vs 21), but with lower output: The 32 Edit calls averaging only 570 tokens each indicate targeted, iterative corrections (mypy fixes, lint, small functional changes) rather than large rewrites. This is a healthy pattern — corrections were small and precise.
  • Bash volume dominated cache reads: 54 Bash calls consumed 5.6M cache read tokens (37.6% of total cache reads). This reflects the cost of piping test suite outputs (594 tests) and tool outputs back through long context. The high call count (54) is justified by the TDD workflow: write tests, run tests, fix, run again.
  • Low rework indicators: Only 1 Agent call (vs 3–4 in v0.3.0) and minimal None-turn reasoning overhead (10 turns vs 17 in v0.3.0 session1, 12 in session2) suggest clean orchestration with little wasted deliberation. The single mypy correction loop and one ruff lint fix are minor.
  • TodoWrite usage dropped: 3 calls (1,292 tokens) vs 10 calls (4,165 tokens) in v0.3.0 session2. Suggests the orchestrator maintained task state more efficiently, likely due to the comprehensive upfront TASKS file rather than ad-hoc todo tracking.
  • Direct input minimal (685 tokens): Only 0.005% of context was non-cached direct input. Nearly all context was served from prompt cache. This is possible because the single-session pattern keeps the system prompt and accumulated context continuously cached.

Methodology and Confidence

All token counts in this report are MEASURED values queried directly from the SQLite telemetry database at C:/Users/Sucea/.claude/usage.db. Session identification used the sessions table filtered by project_name = 'repos/sbom-validator' and cross-referenced against the session timestamps documented in the task brief. Per-tool data was aggregated from the turns table grouped by tool_name.

The following values are inferred or estimated rather than directly measured:

  • (est.) Rework token cost — the DB records which tool was called but does not tag whether a given Edit was corrective vs additive. The rework estimate (low) is inferred from gate evidence: 1 mypy correction loop (3 errors) and 1 lint fix documented in TASKS-v0.4.0.md G3/G4.
  • (est.) Per-gate token attribution — the DB does not record gate boundaries. Gate-level attribution would require structured gate markers in the session.
  • (est.) Subagent-level breakdown — the single Agent call (1,614 output, 350 cache_creation) represents the dispatch overhead. The bulk of work was done in the parent session, so per-agent attribution is not separately measurable from the DB alone.

Confidence: High for all session-level and tool-level metrics. Medium for rework attribution and gate-level estimates.
