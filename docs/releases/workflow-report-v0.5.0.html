5. Top Workflow Gaps

  1. Multi-parser consistency: DEPENDENCY_OF fix applied to 2 of 3 SPDX parsers (F-01).
    Root cause: PR #18 modified spdx_parser.py and spdx_yaml_parser.py to add DEPENDENCY_OF to the qualifying relationship set, but did not modify spdx_tv_parser.py. All three parsers share the same logical relationship-filtering requirement (they produce the same NormalizedSBOM output), but there is no structural enforcement of that parity. The fix was partial because the developer (and reviewer at PR merge time) treated the three parsers as independent files rather than as a coordinated family.
    Impact: Medium. The bug was caught at G4 and fixed before release. However, the fix required a separate developer dispatch (34,389 tokens) and a corrective commit in the release session. If G4 had not been run, this bug would have shipped.
    Fix: Add a multi-parser consistency note to the developer agent definition: when a fix modifies a relationship-handling constant or method in one SPDX parser, verify the same change applies to all SPDX parser variants. Additionally, add a cross-parser consistency check to the reviewer checklist for any SPDX parser changes.
  2. Documentation gaps (F-02/F-03/F-04) reach the release gate rather than being closed at PR merge.
    Root cause: Feature PRs #16, #17, #18 were merged to develop without complete corresponding documentation updates. Specifically: CHANGELOG.md entry for the category field was incomplete (F-02), requirements.md still referenced FR-07 as active after its removal (F-03), user-guide.md did not cover the category field or stable filename changes (F-04). These gaps indicate that documentation is treated as a release-time concern rather than a PR-merge concern.
    Impact: Low-Medium. Three separate documentation findings required a documentation-writer dispatch at 44,208 tokens. If documentation completeness were enforced at PR merge (part of the PR description or reviewer checklist), these findings would not accumulate to the release gate.
    Fix: Add a documentation completeness requirement to the PR merge checklist: any PR that changes behavior, adds a new output field, or modifies/removes an FR must include CHANGELOG, requirements.md, and user-guide.md updates as non-optional PR components. The reviewer at PR merge time should verify these three files are updated before approving the PR.
  3. README.md version example stale after version bump (G7).
    Root cause: The README.md JSON output example block contained tool_version: "0.4.0". After the version was bumped to 0.5.0 at G7, this example became stale. The G7 documentation-writer corrected it, but the issue reflects a pattern: hardcoded version strings in example output blocks are not automatically updated when version changes are applied.
    Impact: Low. Caught at G7 and fixed. But if missed, it would have been a misleading documentation artifact in the released README.
    Fix: Document version-string update locations in the documentation-writer agent checklist for the G7 version bump task: (1) pyproject.toml, (2) __init__.py, (3) CHANGELOG header, (4) README.md example output blocks containing version numbers. This prevents the stale example pattern from recurring.

Overall Trend Verdict

STABLE WITH IDENTIFIED PLATEAU

The sbom-validator workflow has maintained strong structural health across four releases: zero CI failure cycles sustained, 100% gate pass rate, and consistently clean security posture. However, two metrics have plateaued or mildly regressed in v0.5.0: the rework ratio is not declining further (4–8% range for two consecutive releases), and agent compliance dropped to 83% due to documentation and TDD discipline gaps that are rooted in PR-level practices rather than release-level practices.

The primary lever for the next improvement phase is clear: shift documentation completeness requirements upstream from the release gate to the PR merge gate. If CHANGELOG, requirements.md, and user-guide.md updates are enforced at PR merge time, the G4 documentation finding pattern (F-02/F-03/F-04 in v0.5.0, ADR-001 staleness in v0.4.0) will be eliminated. Similarly, the multi-parser consistency check addresses a new failure mode introduced by having multiple structurally related parser files with no shared enforcement mechanism.

The measured token telemetry introduced in v0.5.0 is a meaningful improvement: it provides objective per-agent cost data that was previously unavailable. This enables precise identification of where rework cost is occurring, which directly supports targeted improvements in future releases.
