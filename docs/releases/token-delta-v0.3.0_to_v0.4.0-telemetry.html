Key Structural Differences

1. Single Session vs Two Sessions

v0.3.0 used two sessions across two days (2026-04-13 and 2026-04-14). The second session (7f9e859f) began with 4,035 direct input tokens — the highest non-cached input of any session in the project history. This reflects the context reload cost when resuming after a session boundary: the system prompt, file summaries, and task state had to be re-fed as direct input rather than served from cache. v0.4.0's single-session pattern had only 685 direct input tokens total.

2. Model Mix: Haiku 4.5 in v0.3.0 Session 1

v0.3.0 session 1 (21edb18e) used claude-haiku-4-5, which produced 69,585 output tokens across 171 turns (407 output/turn). The None-turn reasoning in that session was notably high — 11,723 tokens from 17 turns — compared to sonnet-4-6 behavior. v0.4.0 used sonnet-4-6 exclusively (614 output/turn). The model difference affects absolute token counts but also capability: Haiku's lower token-per-turn output reflects shorter, more constrained responses. The two models are not directly comparable on cost without price normalization.

3. Bash Call Reduction (131 vs 54)

The 59% reduction in Bash calls is the single largest structural improvement from v0.3.0 to v0.4.0. v0.3.0 session1 alone had 69 Bash calls, driven by the constants extraction refactoring task which required many iterative test-run cycles. v0.4.0's 54 Bash calls for a larger feature (2 new parsers) indicates more efficient TDD execution — fewer correction cycles per implementation milestone.

4. Agent Call Reduction (7 vs 1)

v0.3.0 used 7 Agent dispatches across its two sessions, including subagents for codebase exploration, constants extraction, and analytics report generation. v0.4.0 used only 1 Agent call, with all implementation work done in the parent orchestrator session. This reduction in agent delegation contributed to the 81% drop in Agent output tokens and reduced subagent context reload overhead.

5. TodoWrite Reduction (10 vs 3)

v0.3.0 session2 (7f9e859f) had 10 TodoWrite calls totaling 4,165 tokens — primarily the release preparation subagent maintaining a task checklist. v0.4.0's 3 calls (1,292 tokens) reflect a leaner task tracking approach, consistent with having a comprehensive TASKS-v0.4.0.md as the primary coordination artifact.

Adopted Improvements and Regressions

Adopted Improvements (v0.3.0 learnings applied in v0.4.0)

  • Single-session orchestration: v0.4.0 completed in one session, eliminating the 4,035-token context reload cost observed at the start of v0.3.0 session2. Direct input tokens fell from 4,803 to 685 (an 85.7% reduction).
  • Fewer subagent dispatches: 1 Agent call vs 7 in v0.3.0. The orchestrator handled more work directly, reducing delegation overhead and subagent context boundaries.
  • Cleaner TDD execution: 54 Bash calls vs 131 — a 59% reduction despite comparable feature scope. The pre-commit hooks (established in v0.2.2 and maintained through v0.3.0) continue to prevent wasted CI cycles at commit time.
  • Less reasoning overhead: None-turn turns fell from 29 to 10 (and from 20,203 to 4,784 output tokens). More direct execution with less mid-session deliberation.
  • Maintained high cache efficiency: 98.1% in v0.4.0 vs 97.3% in v0.3.0. The single-session pattern preserved context warmth throughout the 52-minute session.

Regressions / New Friction Points

  • Write output tokens slightly higher (47,064 vs 44,821): v0.4.0 had more new file creation (2 parsers, 8 fixtures) than v0.3.0 (mainly schema files). This is a justified increase proportional to feature scope, not a regression.
  • Higher output tokens per turn (614 vs 493): Using sonnet-4-6 exclusively vs Haiku/Sonnet mix means individual turns are more verbose. This is a model mix effect, not a workflow regression. Sonnet produces higher quality output but at higher token volume per turn.
  • Skill tool introduced (1 call, 1,767 tokens): A new tool type not present in v0.3.0. Single call suggests intentional one-off use (likely the token-analytics skill), not systematic overhead.
  • Glob calls increased (6 vs 4): Minor. Additional file discovery passes likely for fixture and test file organization.

IMPROVED

v0.4.0 is measurably more token-efficient than v0.3.0 across all key metrics:

  • Total turns: -48.2% (169 vs 326)

  • Cache efficiency: +0.8pp (98.1% vs 97.3%)
  • Bash calls: -59% (54 vs 131)
  • Agent calls: -86% (1 vs 7)

The efficiency improvement is driven by three structural factors: (1) single-session execution eliminating session boundary reload cost, (2) Bash call reduction reflecting cleaner TDD cycles, and (3) reduced agent dispatch overhead. The v0.4.0 feature scope (2 new parsers, 8 fixtures, 78 tests) was at least as complex as v0.3.0 (multi-version schema bundling), making the efficiency gain substantive rather than scope-driven.

Caution: model mix (Haiku in v0.3.0 session1 vs Sonnet-only in v0.4.0) partially inflates the apparent improvement in raw output token reduction. Cost normalization across model pricing would provide a more accurate monetary efficiency comparison, but is beyond the scope of this telemetry-based report.
