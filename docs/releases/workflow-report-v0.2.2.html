  2. Diagnosis: Multiple capture approaches (cmd.exe, PowerShell, Python subprocess) all returned empty output; pattern pointed to Click group never being invoked. Root cause: missing if __name__ == '__main__': main() in cli.py.

  5. Smoke test: New scripts/smoke-test-binary.sh — 14 scenarios verifying output content and exit codes, integrated into release CI.
  6. Process hardening: CI gate added for token report existence; workflow-analyst agent created; release-manager and orchestrator updated.
  7. Release: Tag pushed, GitHub Actions published Linux + Windows binaries. Smoke test: 14/14 passing.

4 — Top Workflow Wins

1
Local poetry run pre-commit hooks — version drift structurally eliminated

Switching from remote-pinned hooks to repo: local + language: system + entry: poetry run means local hooks and CI always use the exact same tool versions. The three CI lint/format failure cycles that occurred in both v0.2.1 and v0.2.2 are now structurally impossible. Estimated saving: ~45,000–60,000 tokens/release from v0.3.x onward.

Token saving: HIGH Quality: HIGHER Effort: LOW
2
14-scenario binary smoke test in CI — broken binary class of bug blocked permanently

scripts/smoke-test-binary.sh verifies output content and exit codes across 14 scenarios (valid/invalid JSON, valid/invalid XML, missing files, error cases). A broken binary like the v0.1.0–v0.2.1 silent exit-0 failure cannot ship again without failing this gate. Integrated into both build-linux and build-windows release jobs.

Quality: CRITICAL Effort: MEDIUM (one-time)
3
Zero session boundary overhead — single continuation avoided ~20,000–30,000 tokens

The v0.2.2 session was a direct continuation of the v0.2.1 session with no context reload. In v0.2.1, two sessions incurred ~20,000–30,000 tokens of context reconstruction cost. While not always achievable, multi-release sessions offer meaningful efficiency when scope allows.

Token saving: MEDIUM
4
CI enforcement gates for G9 (token) and G10 (workflow) — process gaps structurally closed

Both the token report and workflow report are now verified to exist as CI preconditions before the release build jobs run. The pattern (verify-tag-provenance step checking for file existence) is repeatable and low-overhead. Future "forgot to run G9" situations will now be blocked at the CI level rather than discovered post-release.

Quality: HIGHER Effort: LOW

5 — Top Workflow Gaps

GAP
CI-Ops symptom fixing before root-cause fixing (3 rounds before structural fix)
The pre-commit version drift was fixed twice with pin updates before the structural root cause (maintaining two independent version references) was diagnosed. This pattern was essentially identical to v0.2.1 (3 CI rounds). The ci-ops agent definition does not require root-cause analysis before applying the first fix attempt.
Root cause: No escalation path triggers for "same failure category repeating across releases."
Fix: Add "recurring failure across releases → root-cause analysis first" rule to ci-ops agent definition.
GAP
G5 Security deferred for the second consecutive release
xmlschema was added in v0.2.1 without a security review. v0.2.2 added no new dependencies, but G5 was still deferred. The operating model states G5 must complete pre-tag; two consecutive formal deferrals weaken the gate's authority.
Root cause: No release tracker enforces "if D1 is carried from previous release, it must be scheduled in next release tracker at G1."
Fix: Add D1 explicitly to v0.3.x TASKS at G1 planning with security-reviewer as assigned owner.
RISK
G9 and G10 ran post-release (second occurrence for G9)
Token analytics and workflow analytics both ran after the release tag was pushed. The CI enforcement gate was added during the same session, meaning the gate did not block the v0.2.2 release itself.
Root cause: Gate enforcement was designed after the second occurrence rather than after the first.
Fix: Already structurally resolved. CI gate active for v0.3.x onward.
RISK
Planner agent not formally invoked — mid-cycle scope additions were unplanned
Process hardening tasks (pre-commit hooks, CI gates, new workflow-analyst agent) were added reactively during the release cycle rather than as planned scope. This inflated the apparent rework ratio and made the TASKS tracker scope slightly misleading.
Root cause: Patch releases skip the formal G0/G1 planning pass.
Fix: Even for patch releases, a 5-minute planner pass at the start captures process debt items as planned scope.
GAP
Reviewer agent not formally invoked
G4 passed via clean lint output, but no independent reviewer agent pass was recorded. The smoke test exit-code capture bug (set -e + || true issue) was discovered during testing rather than review. A reviewer pass would likely have caught this earlier.
Root cause: Patch releases deprioritize independent review in favour of speed.
Fix: For bug-fix releases touching packaging/binary logic, require a reviewer pass on the changed files before pushing.
