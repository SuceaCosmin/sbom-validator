Top Token Sinks

  1. G3 TDD Build phase — ~235,000 – 256,000 tokens (~40% of total)
    The dominant sink in every new-format-feature release. v0.6.0 introduced a complete new format stack with no code reuse from existing parsers: new detection logic, a new schema validator path (Draft 2020-12 vs. Draft 7), and a new parser module with two-pass graph traversal. Each of the 4 tester dispatches required reading the SPDX 3.0.1 spec and schema before writing tests — a non-trivial orientation cost per dispatch. The 3.F1 parser tester (44 tests, 3 fixtures) was the single largest tester dispatch of the release. The 3 parallel developer implementations in Session 1 are the second-largest contributor within G3. This sink is structurally unavoidable for a new-format release — the cost is proportional to the feature's novelty and complexity, not to inefficiency.
  2. G4 quality review + 4.K1 fix loop — ~100,000 – 126,000 tokens (~18.8% of total)
    The review found 7 mandatory findings. Two arose from a gap between ADR-010's specification and the initial implementation: the envelope-only schema validation approach (M-01/S-M-01) was not explicitly called out in ADR-010 as a deviation from the originally planned full Draft 2020-12 validation — it emerged during implementation when the bundled shacl2code-generated schema rejected minimal valid documents. The registry=Registry() finding (S-M-02) is a Draft 2020-12 API requirement not covered in ADR-010's schema validator design. These findings required a formal deferral (D2), an ADR-010 amendment, and 7 discrete code/doc changes in 4.K1. The rework cost (~45,000–58,000 tokens for 4.K1) is the largest avoidable spend in this release.
  3. G5 security review — ~50,000 – 62,000 tokens (~9.3% of total)
    New file format input parsing is the highest-security-risk surface in the validator. The security reviewer correctly focused on: remote reference fetching prevention (registry= finding), malformed @graph input handling (isinstance guards), and the envelope-only schema validation risk acceptance. The review was efficient — 5 findings, all well-scoped, none requiring repeated investigation. Cost is lower than v0.5.0 G5 because there is no behavioral regression risk (pure addition of a new format) and no external data exposure changes.
  4. G2 Architecture (3 dispatches) — ~45,000 – 55,000 tokens (~8.3% of total)
    Three architect dispatches to evaluate the feature, author ADR-010, and validate the ADR content. The three-dispatch pattern is higher overhead than a single dispatch but reduces post-ADR correction loops. Measured against value: ADR-010 settled R1 (Draft 2020-12 validator), R2 (@context URL), R3 (two-pass traversal) before any implementation, preventing mid-implementation redesign. The one gap ADR-010 left (envelope-only schema deviation not documented) propagated to M-01/S-M-01 in G4/G5 and added ~45,000–58,000 tokens of rework — more than the architecture phase itself cost.
  5. Session boundary context reload — ~30,000 – 50,000 tokens (~6.7% of total)
    The release required 2 sessions because Session 1 hit the 98% token limit before pipeline wiring (3.G2). Session 2 agents each re-read the TASKS tracker, ADR-010, existing modified source files, and test output to orient. This overhead is distributed across multiple early Session 2 dispatches and does not appear in any single gate's budget. It represents pure structural inefficiency from the session boundary — all the actual work done in Session 2 could have been completed without re-reading orientation context if the session had not hit the limit.

Methodology and Confidence

Session totals: Provided by orchestrator as session-level estimates (~280,000 for Session 1, ~320,000 for Session 2, ~600,000 combined). These are labeled EST. The usage.db SQLite database at ~/.claude/usage.db contains exact per-session and per-turn token counts; shell execution was not available in this agent context to query it. The session estimates are consistent with the task scope (19 agent dispatches, 2-session delivery, new format feature of comparable complexity to v0.4.0 which had an estimated total of ~400,000–500,000 tokens before telemetry was available).

Per-agent apportionment: Derived from session totals using the following inputs: (1) task scope and complexity documented in TASKS-v0.6.0.md; (2) subagent identity confirmed from 16 .meta.json files read in session 1a2b056d; (3) relative complexity ratios calibrated against v0.5.0 measured values for comparable task types (reviewer at 79,592, security reviewer at 75,886, developer fix dispatch at 34,389, documentation writer at 44,208); (4) the known asymmetry between tester dispatches (spec-heavy, higher cost) and developer dispatches (implementation, moderate cost) observed consistently across v0.3.0–v0.5.0.

Confidence levels: Session totals: Medium (orchestrator estimates, not DB-measured). Per-agent ranges: Medium-Low (apportioned from session totals with reasonable priors, ±15–25% uncertainty per agent). Rework ratio (~10–15%): Medium (based on 4.K1 task scope and number of findings addressed; confirmed 7 mandatory findings across G4+G5).
