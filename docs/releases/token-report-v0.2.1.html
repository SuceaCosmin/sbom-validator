Top Token Sinks and Repetition Patterns

  1. Schema asset integration and XML validation wiring (~30% of G3 budget)
    The CycloneDX XSD files are large (12,500+ lines combined). Reading, reasoning about, and wiring them into schema_validator.py consumed a disproportionate share of G3 tokens. Partially unavoidable given scope.
  2. CI failure rework cycles — lint/format misalignment (~13–16% of total)
    3 CI failures, all in the same category: code style (ruff import sort + black formatting). Root cause: local checks passed on Windows but CI runs on Linux with slightly different toolchain versions. Each cycle cost ~log fetch + diagnosis + fix commit + push + re-run wait.
  3. Session boundary context reload (~6–9% overhead)
    The work spanned two sessions (Cursor → Claude Code). The resumption required reading a full transcript (~600 lines) plus re-deriving task state from git status. One-time cost but significant.
  4. Iterative UX refinement sandbox loop (~12% of total)
    The presentation layer went through 4–5 incremental feedback iterations (rule IDs removed → humanized paths → hints → separate hint column → SPDX re-test). Each round required re-running the sandbox and reviewing output. Productive but iterative.
  5. Sandbox report cleanup and re-runs
    Multiple sandbox runs accumulated stale reports that required cleanup. Minor but recurring noise per session.

Quality Impact Assessment

The token spend for this release was reasonable relative to scope. The CycloneDX XML feature required genuine complexity: two large XSD assets, dual-mode parser logic, schema_validator rewrite, and full test parity across 7 fixture scenarios. The UX improvements, while iterative, produced measurable end-user value (humanized paths, hint column, cleaner messages).

The main quality risk was the skipped G5 Security gate. The xmlschema dependency was added without a formal security review. For future releases introducing new third-party dependencies, G5 should be mandatory.

Overall token efficiency: ACCEPTABLE — rework ratio (~13–16%) is within normal range for a feature of this size. Primary efficiency opportunity is eliminating preventable CI failures via pre-commit hooks.

Methodology and Confidence

All token values are estimates (EST). No direct API telemetry was available. Estimates were derived from:

  • Observable work artifacts: 37 files changed, ~13,700 lines added/removed
  • Agent loop counts inferred from git commits and CI run history
  • Session boundary overhead estimated from transcript length (~600 lines)
  • CI failure cycle cost estimated from 3 failure runs × average log-fetch-fix-push loop
  • Per-agent estimates based on relative task size and known token density of coding vs documentation tasks

Confidence: Medium. Ranges reflect ±15% uncertainty on each component. Totals should be treated as order-of-magnitude guidance, not precise billing figures.
