4. Top Workflow Wins

  1. Single-session delivery eliminates context reload overhead — measured 98.1% cache efficiency.
    v0.4.0 completed all 11 gates within a single 52-minute session, compared to v0.3.0's two sessions (73 min + 53 min) with a session-boundary context reload between them. The DB shows 98.1% cache efficiency vs 97.3% for v0.3.0 — 0.8pp improvement that translates to fewer tokens written to cache and lower redundant context cost. The single-session pattern kept all context continuously warm across the entire delivery pipeline. MEASURED — cache efficiency from usage.db.
  2. Bash call reduction of 59% confirms fewer test-fix cycles vs v0.3.0.
    v0.4.0: 54 Bash calls. v0.3.0: 131 Bash calls total (two sessions). The -59% reduction is not explained by scope difference alone (v0.4.0 added more tests, +78 vs +15). It reflects cleaner first-pass implementations: rework cycles required fewer re-runs of pytest, ruff, and mypy. Only 1 mypy correction loop and 1 ruff fix required correction vs multiple loops in v0.3.0. MEASURED — Bash call counts from usage.db.
  3. Pre-commit hooks (ruff + ruff-format) sustained zero CI failure cycles for third consecutive release.
    The structural change introduced in v0.2.2 continues to compound value. In v0.4.0: 0 remote CI failures despite adding 2 new parsers, 8 fixture files, and 3 new test modules. Pre-commit caught 1 lint error (unused imports) and 6 format corrections at commit time, preventing remote CI pipeline waste for a zero-overhead fix. (inferred) from TASKS-v0.4.0.md G6 evidence + Bash call count.
  4. Agent dispatch count dropped -86% (1 vs 7) — tighter orchestration with less subagent overhead.
    v0.4.0 dispatched 1 Agent call vs 7 in v0.3.0 (across two sessions). This reflects a more direct orchestration approach where the orchestrator ran all specialist work inline rather than spawning separate subagent contexts. Per-subagent overhead (context initialization, system prompt load per subagent) was minimized. The 1 Agent call generated only 1,614 output tokens and 350 cache_creation tokens — the dispatch cost itself was negligible. MEASURED — Agent call count from usage.db.

5. Top Workflow Gaps

  1. Write tool is the top output sink at 45.4% — high file-creation overhead is inherent but partially reducible.
    Root cause (measured): 21 Write calls generated 47,064 output tokens — 45.4% of the session total. This is characteristic of a two-new-parser release: spdx_tv_parser.py, spdx_yaml_parser.py, 8 fixture files, 3 test files, ADR-009, and multiple doc updates all required full Write calls. Average 2,241 tokens per Write call.
    Impact: Medium. Large Write calls are inherent to new-module releases, but 8 fixture files could potentially be consolidated into fewer Write calls (batch 4 TV files in one reasoning step) or generated via a helper script.
    Fix: For releases adding 4+ fixture files of similar structure, consider a generator script rather than individual Write calls. Estimated reduction: 5–8 fewer Write calls, ~10,000–15,000 fewer output tokens. MEASURED — Write call count and output tokens from usage.db.
  2. Zero Grep calls in the entire session — code navigation relies exclusively on Read + Glob.
    Root cause (measured): The DB shows 0 Grep calls in session b1fcffcb, while 39 Read calls and 6 Glob calls were made for navigation. For a session adding 2 new parsers that extend existing modules (format_detector.py, schema_validator.py, validator.py), finding the exact insertion points without Grep likely required reading entire files rather than targeted pattern searches.
    Impact: Low-medium. Read tool generated 107,129 cache creation tokens vs the much lower cost of targeted Grep + Read. Over-reading (reading full files when only a function signature is needed) inflates cache creation cost.
    Fix: The orchestrator/developer should prefer Grep for function signature lookup before reading full files. A Grep for "def detect_format" or "def validate_schema" provides the exact line context without a full file Read. MEASURED — Grep call count (0) from usage.db.
  3. All agent work consolidated in 1 subagent dispatch — specialist agent boundaries are invisible in telemetry.
    Root cause (measured): 1 Agent call vs 7 in v0.3.0. While this improves execution efficiency (fewer context reloads), it eliminates the ability to attribute token cost to specific specialist roles (Tester, Developer, Reviewer, etc.). The DB records tool usage at the session level, not by specialist agent boundary.
    Impact: Medium for reporting quality. This workflow report and the token report cannot provide per-agent cost breakdowns — only session-level aggregates and gate-level inferences.
    Fix: Add structured gate markers (e.g., TodoWrite entries with gate IDs at each G-transition) to allow token attribution by gate even within a single subagent session. This does not require separate Agent dispatches — it adds negligible overhead (~3 Bash calls, ~300 tokens) while enabling the DB to be post-processed by gate timestamp. MEASURED — Agent call count from usage.db + inference about DB structure.

Overall Trend Verdict

IMPROVING — supported by measured telemetry data for the first time

The sbom-validator project workflow is on a measurable improvement trajectory. Unlike prior releases where "improving" was an inferred judgment from qualitative gate evidence, v0.4.0 provides measured evidence across five quantitative dimensions: output tokens down 35.4%, turns down 48.2%, Bash calls down 59%, Agent dispatches down 86%, and cache efficiency up 0.8pp — all while delivering larger scope (+78 tests, +2 parsers, +1 ADR vs v0.3.0's +15 tests, +0 parsers, +0 ADR).

The structural improvements introduced in v0.2.2 (pre-commit hooks) continue to compound. Single-session orchestration is now established as the preferred execution pattern for releases completing within 60 minutes. The two remaining structural gaps — lack of gate boundary markers in telemetry, and ADR cross-reference staleness — are both low-effort to address and have no impact on current release quality.

Primary area to watch for v0.5.0: whether the Write-tool-dominant cost pattern (45.4% of output) persists if additional format parsers are added. If v0.5.0 adds a third parser format (e.g., SPDX RDF), a fixture generator script would be the highest-leverage intervention to prevent Write overhead from scaling linearly with format count.
