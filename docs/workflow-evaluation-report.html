Human-in-the-Loop AI Workflow — Evaluation Report

Post-project analysis of the multi-agent Specification Driven Development workflow used to build sbom-validator v0.1.0. Covers agent maturity, token-efficiency, repetition patterns, and concrete improvement ideas.

📅 Session date: 2026-04-06 🚀 Deliverable: sbom-validator v0.1.0 🤖 Model: Claude Sonnet 4.6 (Pro subscription)

Executive Summary

The session successfully delivered a production-quality CLI tool — 6 modules, 2 parsers, 2 bundled schemas, 358 tests, 97% coverage, zero static-analysis errors, full documentation, and a tagged GitHub release — in a single working session using a 6-agent multi-agent workflow.

The workflow was effective but not yet efficient. The Pro subscription 4-hour rate limit was reached comfortably within the session, primarily because of high per-agent cold-start costs and several avoidable rework cycles. The quality bar set by TDD + code review was the right call and contributed directly to catching 3 critical architectural bugs before release. The opportunity is to reach the same quality with significantly fewer tokens.

Verdict

🔴 Inefficient — specifically in Phase 2 (core implementation)

Approximately 35–40% of total token usage (est.) was attributable to Phase 2's 14 agent invocations, each independently loading 4–6 shared context files. The files read repeatedly included requirements.md (~3K tokens), normalized-model.md (~2K tokens), 5 ADR files (~8K tokens combined), and models.py (~1.5K tokens) — roughly 14,500 tokens of duplicated input per agent, or ~200K tokens of repeated context loading across Phase 2 alone.

An additional ~80–100K tokens (est.) were consumed by avoidable rework: the invalid-schema fixture fix, the ADR-001/002 violation remediation in the code review, and the three CI failure rounds.

That said, the token spend produced a high-quality artifact. A less thorough workflow (no TDD, no code review) would have shipped the 3 critical bugs. The question is not whether to spend tokens on quality gates — it is how to front-load quality signals to avoid expensive post-hoc correction.

4 — Improvement Ideas

The following ideas target token efficiency while maintaining or improving the current output quality. Each idea includes an estimated token saving, implementation effort, and quality impact.

1
Create a Compact "Project Briefing" File as Shared Agent Context

Currently every agent starts cold and reads requirements.md, normalized-model.md, 5 ADRs, and models.py to build its mental model of the project. These files total ~15,000 tokens and are read ~14 times across the session.

Create a single docs/agent-briefing.md file (~500 tokens) that distills only the decision-critical facts each agent needs: the 5 ADR decisions in 1–2 sentences each, the NormalizedSBOM field contract (the table, not the prose), the FR-to-NTIA element mapping table, and the exact function signatures from each module. Every agent prompt includes this file instead of reading the originals.

Rational: Agents need the decisions, not the rationale. The rationale is valuable for humans reviewing the ADRs but is wasted tokens for an agent that only needs to know "parsers accept dict, not Path" before coding.
💰 Est. saving: ~180K tokens (Phase 2 context loading) ✅ Quality: neutral or improved (less noise) 🔧 Effort: 1–2 hours to write
2
Add a Mandatory Pre-Handoff Lint Gate to Developer and Tester Agents

Three CI failure rounds occurred because agents declared tasks done without running the full static analysis suite locally. The Developer agent's "before marking done" checklist exists but was not enforced. The Tester agent has no similar checklist at all.

Update both agent definitions to include a non-negotiable terminal step: "Run poetry run ruff check src/ tests/ && poetry run black --check src/ tests/ && poetry run mypy src/. If any command exits non-zero, fix the issues before reporting done. This step is mandatory — it is not done until all three pass." Elevate this to the top of the file so it appears early in the agent's context window.

Rational: Each CI failure round cost approximately one agent invocation to diagnose + fix (read logs, read source, edit, commit, push, wait). Running lint locally costs ~1 Bash tool call and a few hundred tokens. The cost ratio is 100:1 in favor of local lint.
💰 Est. saving: ~80–100K tokens (CI fix cycles) ✅ Quality: improved (catches issues earlier) 🔧 Effort: 30 min (edit 2 agent files)
3
Add Concrete Type Signatures (Runnable Python Stubs) to ADRs

Two of the three CRITICAL code-review findings (R-01/R-02: version guards missing; R-04/R-05: parser/detector signature mismatch) were caused by ADRs that described the interface in prose rather than in code. The Developer agent implemented plausible but incorrect interfaces.

Each ADR that defines a module interface should include a Python stub block: 

# ADR-001 contract — format_detector.py
def detect_format(document: dict[str, Any]) -> Literal["spdx", "cyclonedx"]:
    """Raises UnsupportedFormatError if format is unrecognized or wrong version."""
    ...
The Developer agent's definition should require: "Before implementing any module, read its ADR stub and verify your function signatures match exactly."

Rational: The code review (Phase 4) cost ~150K tokens to run and produced 13 findings, of which 7 required a Developer fix pass (~100K tokens). If the ADR stubs had prevented the 2 signature violations, the review would have found only 11 issues and the fix pass would have been shorter. Preventing one CRITICAL finding upstream saves the cost of the entire downstream correction chain.
💰 Est. saving: ~50–70K tokens (review + fix) ✅ Quality: improved (enforced contracts) 🔧 Effort: 2–3 hours (update 5 ADRs)
4
Use Targeted Test Runs Instead of Full Suite Runs During Development

The full test suite (358 tests, ~6–7s) was run after every individual module implementation. During Phase 2, this means 7 full runs × 358 tests, most of which test modules not yet changed. The tests themselves are fast, but reading and processing the full pytest output (~3K tokens per run) adds up.

Update the Developer and Tester agent definitions to use targeted pytest invocations during implementation: poetry run pytest tests/unit/test_<module>.py -v. Run the full suite only once per phase as the final gate, not after each module.

Rational: A targeted run produces ~200 tokens of output vs. ~3,000 for a full run. Across 7 development phases × 2–3 intermediate runs each, this saves approximately 35–50K tokens in pytest output processing. It also gives faster feedback, which reduces the cost of fix-and-retry loops.
💰 Est. saving: ~35–50K tokens ✅ Quality: neutral (same coverage at phase-end) 🔧 Effort: 30 min (edit 2 agent files)
5
Introduce a Planner "Scope Lock" Step Before Each Phase

Several scope ambiguities caused invisible token waste during the session: "implement schema validator" silently included downloading and bundling 2 JSON schemas (which failed and needed a separate recovery agent); "create invalid-schema fixtures" was ambiguous about whether the format must still be recognizable. These ambiguities caused downstream rework.

Before dispatching agents for a phase, invoke the Planner agent with a single instruction: "For each task in the upcoming phase, list: (a) every file that will be created or modified, (b) every external resource that will be needed (schemas, downloads, network), and (c) every assumption this task makes about the current state of the codebase. Flag any assumption that is not yet verified." The user reviews this output as the phase approval gate — catching scope issues before tokens are spent implementing them.

Rational: The Planner agent is the most underutilized agent in the workflow. Its current definition lists good task attributes (inputs, outputs, acceptance criteria) but it was never actually called as a standalone agent — planning happened inline by the orchestrator. Using the Planner as a structured scope-lock step adds ~20–30K tokens per phase but likely prevents 1–2 rework cycles per project, each of which costs 50–100K tokens.
💰 Net est. saving: ~100K tokens across a project ✅ Quality: improved (fewer surprises) 🔧 Effort: 2–3 hours (rewrite Planner agent + workflow)

5 — What Worked Well (Keep Doing)

🧪 TDD was non-negotiable

Writing tests before implementation consistently. The Tester agent never wrote tests against stubs. This discipline directly prevented several implementation shortcuts.

🔍 Code review before release

The Phase 4 Reviewer agent found 3 CRITICAL bugs that would have shipped. The review was expensive but paid for itself in output quality. Never skip this gate.

⚡ Parallel agent dispatch

Running Tester + Developer agents for independent tracks (SPDX, CycloneDX, format detector) in parallel dramatically compressed wall-clock time without quality loss.

📋 TASKS.md as living tracker

Updating TASKS.md after every task was the right call. When the context was compacted and a new session started, TASKS.md enabled immediate resumption without re-reading conversation history.

🏛️ Phase approval gates

Human approval between phases kept the Human-in-the-Loop meaningful. Without these gates, the workflow would be autonomous execution with no oversight points.

📄 Spec-first development

Writing requirements and ADRs before any code gave the Developer agent a ground truth to implement against. Without this, agents tend to make their own architectural decisions inconsistently.
