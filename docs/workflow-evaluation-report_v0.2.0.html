4 — New Issues Emerging in v0.2.0

Issues that did not exist in v0.1.0 and were not anticipated by the improvement plan.

Issue A — Bash Tool Permission Blocking Pattern

Three agents were blocked from running git commit via Bash, requiring manual orchestrator intervention. This was not a problem in v0.1.0 (different session or permission scope). The pattern is: agent completes all file writes successfully, then fails at the commit step, returns a "here are the commands to run manually" response. This is recoverable but costs orchestrator attention and 1–2 additional agent invocations.

Recommendation: Add a standard "blocked commit fallback" section to all agent definitions: "If git commit is blocked by tool permissions, output the exact commands in a code block and mark the task as incomplete — the orchestrator will run them." This makes the failure mode explicit and handled rather than producing an ambiguous half-done state.

Issue B — PyInstaller Transitive Dependency Data Files

PyInstaller does not automatically bundle non-Python data files from transitive dependencies (files ending in .lark, .json, etc.). rfc3987_syntax (a dependency of jsonschema) ships a .lark grammar file that is loaded at runtime. The ADR-008 risk register focused on direct dependencies (spdx-tools, cyclonedx-bom) but did not inspect the transitive dependency graph for data assets.

Fix applied: Added collect_data_files('rfc3987_syntax') and collect_data_files('jsonschema') to the spec file. This is the correct PyInstaller idiom. The fix was straightforward once the error was identified from CI logs.

Issue C — GitHub Actions Windows + snok/install-poetry@v1 PATH Issue

snok/install-poetry@v1 does not reliably add Poetry to PATH in the PowerShell shell used by Windows GitHub Actions runners. This is a known limitation documented in the action's issues. The fix (pip install poetry + shell: bash) is simple but was not pre-empted in ADR-008.

Recommendation: Add to the Architect agent definition or to a project-level CI conventions note: "On Windows GitHub Actions runners, use pip install poetry and shell: bash as defaults. Avoid snok/install-poetry@v1 on Windows due to PATH propagation issues in PowerShell."

5 — Improvement Ideas for Next Work Package

1
Add a "Blocked Bash Fallback" Section to All Agent Definitions

Three agents were blocked from git commit in this session. The pattern is consistent and predictable. Each agent definition should include a terminal section: "If any Bash command is denied by tool permissions: output the exact commands in a fenced code block, state clearly that the task is incomplete pending orchestrator action, and do not mark the task done." This prevents the ambiguous half-done state where an agent returns a success-sounding summary but the commit never happened.

💰 Est. saving: ~20–30K tokens (3× follow-up agents eliminated) ✅ Quality: improved (no missed commits) 🔧 Effort: 30 min (add section to 5 agent files)
2
Add PyInstaller Data File Checklist to Architect Agent

For any ADR involving binary distribution, the Architect agent should be required to enumerate non-Python data assets in the dependency tree. The check is simple: "For each runtime dependency, inspect its installed directory for non-.py files. Any .json, .lark, .yaml, .txt or similar files that are loaded at runtime must be explicitly listed in datas using collect_data_files(pkg)." This one check would have prevented the Linux binary failure entirely.

💰 Est. saving: ~30–50K tokens (1 release re-run + fix cycle eliminated) ✅ Quality: improved (first-pass binary works) 🔧 Effort: 1 hour (update Architect definition + add to ADR template)
3
Introduce a Lightweight "Spot Review" for New Modules (not full Phase 4)

v0.2.0 had no dedicated code review pass. v0.1.0's full Phase 4 review found 13 findings including 3 CRITICAL bugs. The right pattern for a maintenance work package (new features on an existing, reviewed codebase) is a lightweight spot review: the Reviewer reads only the new modules (logging_config.py, report_writer.py) and the modified cli.py against the relevant ADRs. This costs ~15–25K tokens vs. ~150K for a full review, and would have caught the frozen-binary issue if included.

The Planner should explicitly decide per-work-package: full review (new core logic), spot review (new features on existing base), or skip (docs/config only).

💰 Token cost: +15–25K tokens (new cost, but prevents larger rework) ✅ Quality: improved (catches issues before release) 🔧 Effort: 1 hour (add spot review task to Planner template)
4
Add Planner Output Length Constraint

The Planner consumed ~32K tokens producing a very detailed plan document with extensive prose per task. The orchestrator primarily needs: task table (ID, agent, deps, deliverables), dependency graph, risk register (top 5). Add to the Planner definition: "Output the plan as: (1) summary table — max 20 rows, (2) dependency graph — ASCII or list format, (3) top 5 risks — 2 sentences each. Total output should not exceed 800 words of prose."

💰 Est. saving: ~15–20K tokens per planning invocation → Quality: neutral (same decisions, less prose) 🔧 Effort: 20 min (edit Planner definition)
5
Revisit 100% Coverage Target — Return to 97% with Named Exceptions

The quality gate agent consumed ~78K tokens — the highest single-agent spend in the session — largely due to the 100% coverage target requiring tests for frozen-binary paths, PackageNotFoundError fallbacks, and deep exception branches. These are legitimate test cases but cost disproportionately. A policy of 97% minimum with explicit # pragma: no cover annotations for genuinely untestable paths (e.g., the sys._MEIPASS frozen binary branch) would reduce the quality gate cost by an estimated 20–30% while maintaining meaningful coverage.

💰 Est. saving: ~15–25K tokens at quality gate → Quality: acceptable (named exceptions are documented) 🔧 Effort: 30 min (policy decision + add 3–4 pragma annotations)
